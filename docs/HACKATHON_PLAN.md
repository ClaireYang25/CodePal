# Hackathon 冲刺计划 (Pre-Deadline Polish Plan)

本文档定义了在 Hackathon 截止日期前，我们需要完成的最后冲刺任务。**核心原则是：不再引入任何新的、高风险的核心功能，专注于稳定、打磨和展示。**

---

## 1. 核心任务：回归测试与稳定性加固 (Regression Testing & Stabilization)

**目标**: 确保所有已实现的功能在各种边界条件下都能稳定工作，杜绝在演示过程中出现意外 Bug。

-   [ ] **全功能回归测试**:
    -   [ ] **新用户首次安装**: 模拟新用户，确保插件即装即用，流程顺畅。
    -   [ ] **OTP 提取（核心链路）**:
        -   [ ] **简单邮件**: 测试标准格式的 OTP 邮件（如 “Your code is 123456”）。
        -   [ ] **复杂邮件**: 测试包含多个数字（订单号、航班号）的干扰邮件。
        -   [ ] **多语言**: 测试中文、英文等不同语言的 OTP 邮件。
        -   [ ] **短验证码**: 测试位数小于 4 的 OTP，确认会被启发式规则正确拒绝（符合预期）。
    -   [ ] **“半后台”监控**: 保持 Gmail 标签页在后台，确认新邮件到达后能自动提取并更新 Popup。
    -   [ ] **主动感知**: 在 Vercel/Notion 等网站的登录页，确认聚焦 OTP 输入框后，Service Worker 能收到“高警戒”信号。
    -   [ ] **自动呈现**: 确认提取成功后，Popup 会自动弹出，或在失败时有系统通知作为兜底。
    -   [ ] **静默机制**: 确认在浏览已读邮件、非 OTP 邮件时，插件保持静默，不会触发任何提取。

-   [ ] **错误处理审查**:
    -   [ ] 逐一检查 `console.warn` 和 `console.error` 日志，确保所有可预见的异常（如网络错误、API 限制）都得到了妥善处理，不会导致插件崩溃。

## 2. 重点任务：用户体验打磨 (UX Polish)

**目标**: 提升所有用户可见界面的质感和交互流畅度，给评委和用户留下专业、完整的印象。

-   [ ] **Popup UI/UX 优化**:
    -   [ ] **加载状态**: 当 Popup 打开，正在从 `storage` 读取最新 OTP 时，增加一个优雅的加载动画（loading spinner）。
    -   [ ] **空状态**: 当从未提取到 OTP 时，显示一个友好的引导页，告知用户“等待新的验证码邮件...”。
    -   [ ] **时间戳格式**: 将“提取时间”从绝对时间戳转换为更人性化的相对时间（如，“刚刚”、“30秒前”）。
    -   [ ] **一键复制**: 为显示的 OTP 数字增加一个明显的“点击复制”功能，并提供复制成功的视觉反馈。

-   [ ] **系统通知交互增强**:
    -   [ ] 为弹出的系统通知增加一个“复制验证码”的按钮，让用户可以直接从通知中获取 OTP，而无需打开 Popup。

## 3. 收尾任务：文档与演示准备 (Documentation & Presentation)

**目标**: 准备好向评委和公众清晰、高效地展示项目亮点所需的所有材料。

-   [ ] **完善 `README.md`**:
    -   [ ] 更新项目截图和 GIF 动图，展示最新的 UI 和核心功能。
    -   [ ] 简洁明了地介绍项目解决了什么问题、核心亮点是什么（三层引擎、半后台、主动感知）。
    -   [ ] 提供清晰的安装和使用指南。

-   [ ] **创建 `USAGE_GUIDE.md`**:
    -   [ ] 撰写一份简单的用户手册，解释插件的各项功能和最佳实践（例如，“保持一个 Gmail 标签页以获得最佳体验”）。

-   [ ] **准备演示材料**:
    -   [ ] 撰写一个 1-2 分钟的演示脚本，覆盖从“请求验证码”到“自动呈现”的完整流畅体验。
    -   [ ] 提前录制一个完美的演示视频，以防现场网络或环境问题。

---
**结论**: 在 Deadline 之前，我们将严格遵守此计划，不再偏离。我们的目标是在现有坚实功能的基础上，交付一个稳定、精致、令人印象深刻的作品。
